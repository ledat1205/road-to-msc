Core Concepts & Architecture

What is Apache Kafka?
⚡Definition

Distributed, fault-tolerant, high-throughput publish-subscribe messaging system for real-time data streaming. Originally developed at LinkedIn.

🎯Core Use Cases
  • Event streaming & real-time pipelines
  • Log aggregation & metrics
  • Change Data Capture (CDC)
  • Stream processing & data integration
Component Reference
ComponentRoleKey Facts
BrokerServer storing & serving messagesIdentified by broker.id; handles all I/O
ProducerPublishes messages to topicsPush model; batches messages for throughput
ConsumerSubscribes to and reads messagesPull model; tracks offset position
TopicNamed feed / category for messagesLogical grouping of events
PartitionOrdered, immutable sequence of recordsUnit of parallelism in Kafka
ZooKeeper / KRaftCluster coordination & metadataKRaft replaces ZooKeeper in 3.x+
Consumer GroupGroup of consumers sharing loadEach partition consumed by one member only
OffsetPosition of a record in a partitionMonotonically increasing integer per partition
ISRIn-Sync Replicas fully caught up to leaderNew leader elected from ISR on failure
ControllerBroker managing leader electionsExactly one per cluster at all times
Topic & Partition Deep Dive
ConceptExplanation
PartitionEach topic is split into N partitions. Each partition is an ordered, immutable append-only log.
Replication FactorNumber of copies of each partition across brokers. Recommended: 3 for production.
Leader PartitionHandles all reads/writes for a partition. Elected by ZooKeeper/KRaft.
ISRSet of replicas fully caught up with the leader. If leader fails, new leader elected from ISR.
Log Retentionretention.ms or retention.bytes controls how long/how much data is kept. Default: 7 days.
Log CompactionKeeps only the latest value per key. Set log.cleanup.policy=compact. Used for KTable topics, CDC.
SegmentEach partition is divided into segment files. Default segment size: 1 GB.
💡 Interview Tip Know the difference between partition (physical log file) and topic (logical group). More partitions = more parallelism, but also more overhead (file handles, replication traffic, leader election time).

Partitioning Strategy
  • Key specified: murmur2(key) % numPartitions — same key always goes to same partition (ordering guarantee)
  • No key: Sticky partitioner (Kafka 2.4+) — batches to one partition, then switches
  • Custom: Implement Partitioner interface for custom routing logic
Java Producer Pattern
Java
Properties props = new Properties();
props.put(ProducerConfig.BOOTSTRAP_SERVERS_CONFIG, "broker1:9092");
props.put(ProducerConfig.KEY_SERIALIZER_CLASS_CONFIG, StringSerializer.class);
props.put(ProducerConfig.VALUE_SERIALIZER_CLASS_CONFIG, StringSerializer.class);
props.put(ProducerConfig.ACKS_CONFIG, "all");
props.put(ProducerConfig.ENABLE_IDEMPOTENCE_CONFIG, true);
props.put(ProducerConfig.COMPRESSION_TYPE_CONFIG, "snappy");
props.put(ProducerConfig.LINGER_MS_CONFIG, 20);
props.put(ProducerConfig.BATCH_SIZE_CONFIG, 65536);

KafkaProducer<String, String> producer = new KafkaProducer<>(props);
ProducerRecord<String, String> record = new ProducerRecord<>("my-topic", "key", "value");
producer.send(record, (metadata, exception) -> {
    if (exception == null)
        System.out.println("Sent to partition " + metadata.partition() + " offset " + metadata.offset());
    else exception.printStackTrace();
});
producer.flush(); producer.close();

Consumers & Consumer Groups

Consumer Group Mechanics
Partition Assignment Rules
  • Each partition → exactly ONE consumer per group
  • Multiple groups can read the same topic independently
  • Active consumers ≤ partition count (extras idle)
  • Group Coordinator: broker managing membership
  • Group Leader: first consumer to join; assigns partitions
Rebalance Triggers
  • Consumer joins or leaves the group
  • Consumer crashes / heartbeat timeout
  • Topic partition count changes
  • Subscription pattern change
  • Poll interval exceeded (max.poll.interval.ms)
Critical Consumer Configuration

Eager< 2.4Stop-the-World — All consumers stop, release all partitions, rejoin. High latency spike.

Python Consumer Pattern
Python
from confluent_kafka import Consumer, KafkaError

conf = {
    'bootstrap.servers': 'broker1:9092',
    'group.id': 'my-consumer-group',
    'auto.offset.reset': 'earliest',
    'enable.auto.commit': False,
    'max.poll.interval.ms': 300000
}
consumer = Consumer(conf)
consumer.subscribe(['my-topic'])

try:
    while True:
        msg = consumer.poll(timeout=1.0)
        if msg is None: continue
        if msg.error(): handle_error(msg.error()); continue
        process(msg.value().decode('utf-8'))
        consumer.commit(asynchronous=False)  # manual commit after processing
finally:
    consumer.close()  # always close to trigger final sync commit

Delivery Semantics & Exactly-Once

The Three Delivery Guarantees
SemanticProducerConsumerRisk
At-Most-Once acks=0, retries=0 Commit before processing Data loss
At-Least-Once acks=all, retries=MAX Commit after processing Duplicates
Exactly-Once (EOS) enable.idempotence=true + transactions isolation.level=read_committed Complex setup
How EOS Works
  • Idempotent Producer: Each producer gets PID + sequence number. Broker deduplicates retries automatically.
  • Transactions: Group multiple topic/partition writes into atomic units using transactional.id.
  • Transaction Coordinator: Special broker partition managing transaction state (begin/commit/abort).
  • Consumer Isolation: read_committed ignores uncommitted/aborted records.
  • Kafka Streams EOS: Set processing.guarantee=exactly_once_v2 — built-in, no manual work.
Java — Transactions
props.put("enable.idempotence", "true");
props.put("transactional.id", "txn-id-001");
producer.initTransactions();
producer.beginTransaction();
try {
    producer.send(record1);
    producer.send(record2);
    producer.sendOffsetsToTransaction(
        offsets, consumerGroupMetadata);
    producer.commitTransaction();
} catch (ProducerFencedException e) {
    producer.close();
} catch (KafkaException e) {
    producer.abortTransaction();
}

Kafka Streams

Core Abstractions
AbstractionSemanticsBacked By
KStreamUnbounded event stream. Each record = independent insert.Source topic partitions
KTableChangelog / materialized view. Latest value per key = upsert.State store + changelog topic
GlobalKTableFull copy on ALL stream instances. Broadcast join/lookup.State store (no repartition)
StateStoreLocal RocksDB store for stateful operations.Backed by compacted changelog topic
💡 Key Distinction KStream join requires repartition if keys change. GlobalKTable avoids repartition — full table on every instance. Use GlobalKTable for small lookup tables; KTable for large, partitioned state.

Windowing Types
Tumbling Window

Fixed-size, non-overlapping. Every record falls in exactly one window. e.g., hourly reports.

TimeWindows.ofSizeWithNoGrace(Duration.ofMinutes(5))
Hopping Window

Fixed-size, overlapping. Advance < size. Records can be in multiple windows. e.g., 5-min window, 1-min hop.

TimeWindows.of(Duration.ofMinutes(5)).advanceBy(Duration.ofMinutes(1))
Session Window

Variable-size based on activity gaps. Merges windows within inactivity period. e.g., user sessions.

SessionWindows.ofInactivityGapWithNoGrace(Duration.ofMinutes(30))
Sliding Window

Created based on record timestamps. Window defined by maxTimeDifference between records. e.g., per-record context.

SlidingWindows.ofTimeDifferenceWithNoGrace(Duration.ofMinutes(5))
Kafka Streams Code Example
Java — Word Count with Windowing
StreamsBuilder builder = new StreamsBuilder();

KStream<String, String> events = builder.stream("input-topic");

KTable<Windowed<String>, Long> wordCounts = events
    .flatMapValues(v -> Arrays.asList(v.toLowerCase().split("\\s+")))
    .groupBy((k, word) -> word)
    .windowedBy(TimeWindows.ofSizeWithNoGrace(Duration.ofMinutes(1)))
    .count(Materialized.as("word-count-store"));

wordCounts.toStream()
    .map((window, count) -> new KeyValue<>(window.key(), count.toString()))
    .to("output-topic", Produced.with(Serdes.String(), Serdes.String()));

Properties props = new Properties();
props.put(StreamsConfig.APPLICATION_ID_CONFIG, "word-count-app");
props.put(StreamsConfig.BOOTSTRAP_SERVERS_CONFIG, "broker:9092");
props.put(StreamsConfig.PROCESSING_GUARANTEE_CONFIG, StreamsConfig.EXACTLY_ONCE_V2);

KafkaStreams streams = new KafkaStreams(builder.build(), props);
streams.start();

Schema Registry & Serialization

Compatibility Modes
ModeDescriptionSafe Changes
BACKWARD (default) New schema can read data written with previous schema Delete fields, add optional fields with defaults
FORWARD Previous schema can read data written with new schema Add fields, delete optional fields
FULL Both backward and forward compatible Only add/remove optional fields with defaults
BACKWARD_TRANSITIVE Compatible with ALL previous versions (not just last) Strictest backward; best for long-term safety
NONE No compatibility checks Any schema change (use only in dev)
💡 Wire Format Every message serialized with Schema Registry has a 5-byte prefix: [Magic Byte 0x00] [4-byte Schema ID] followed by the Avro/Protobuf payload. Consumers use the schema ID to look up the correct schema dynamically.

Top Interview Questions & Answers

Click to expand answers ▾

Why is Kafka so fast? What makes it performant?▼
Sequential disk I/O — Kafka appends to log files sequentially; modern disks do sequential writes at 600MB/s+ vs ~100MB/s random. Zero-copy — uses Linux sendfile() syscall to transfer data directly from page cache to network socket, bypassing user space. Batching — producers/consumers batch messages, amortizing per-message overhead. Compression — snappy/lz4/zstd reduce network I/O. OS Page Cache — Kafka defers to the OS page cache rather than managing its own in-memory cache.
How does Kafka guarantee message ordering?▼
Ordering is guaranteed per partition, not globally across a topic. To guarantee global ordering: use 1 partition (sacrifices parallelism). For per-key ordering: use message keys — same key always routes to same partition via murmur2(key) % numPartitions. With idempotent producer, set max.in.flight.requests.per.connection=5 (safe with idempotence); without it, set to 1.
What happens when a Kafka broker fails?▼
1. ZooKeeper/KRaft detects broker failure (session timeout). 2. Controller broker receives notification. 3. Controller elects new leaders for all partitions where the failed broker was leader — choosing from the ISR. 4. Controller propagates updated metadata to all brokers. 5. Producers/consumers get NotLeaderForPartitionException, fetch new metadata, and reconnect to new leaders automatically. Data loss only occurs if leader fails before ISR replication AND unclean.leader.election=true.
Explain the difference between KStream, KTable, and GlobalKTable.▼
KStream: Event stream — each record is an independent insert event. Think of a stream of clicks or transactions. KTable: Changelog stream (materialized view) — each record is an upsert; latest value per key wins. Think of a user profile table. Partitioned like the source topic. GlobalKTable: Full table replicated on ALL instances. No repartition needed for joins. Use for small lookup tables. Key join difference: KTable join uses co-partitioned data (requires same partition count + key serializer); GlobalKTable is always available locally.
How do you choose the right number of partitions?▼
Formula: max(target_throughput / producer_throughput_per_partition, target_throughput / consumer_throughput_per_partition). Rule of thumb: ~10MB/s write per partition, ~50MB/s read. Also consider: max consumer parallelism needed, retention period × throughput × replication factor for disk estimate, number of brokers (partitions should spread evenly). You can increase partitions but never decrease — plan upfront. Too many partitions: more file handles, slower failover, more memory.
What causes consumer lag and how do you fix it?▼
Causes: slow message processing, insufficient consumer parallelism, complex business logic, external I/O calls (DB lookups), GC pauses, rebalances. Fixes: 1. Increase partition count + scale consumer group proportionally. 2. Optimize processing (async I/O, batching). 3. Reduce max.poll.records if processing takes too long. 4. Horizontal scaling. 5. Use Kafka Streams with state stores for complex logic instead of external lookups. 6. Profile and remove bottlenecks in the processing pipeline.
Explain log compaction and when to use it.▼
Log compaction keeps at least the latest value for each message key, removing older records with the same key. Set log.cleanup.policy=compact. Kafka runs a background log cleaner thread that merges log segments. Use cases: KTable changelog topics, CDC (latest DB row state), user preference stores, any scenario needing "latest state" semantics. NOT for: append-only event logs, audit trails, time-series data where every event matters. Null value = tombstone (delete the key from the log after compaction).
Kafka vs traditional message queues (RabbitMQ, SQS)?▼
Kafka: Distributed log. Messages persist after consumption. Multiple consumer groups read independently. Supports replay. Partitioned for parallelism. Optimized for throughput (millions/sec). Pull-based. Great for event streaming, event sourcing, CDC. Traditional MQs: Delete messages after consumption (typically). One consumer per message. Lower throughput. Push-based. Better for task queues, RPC, complex routing. Choose Kafka when you need: multiple consumers, replay, high throughput, event sourcing, durable pub/sub.
Design a real-time ETL pipeline using Kafka.▼
Architecture: Source DB → Debezium CDC (Connect Source) → Raw Kafka Topic → Kafka Streams (transform/enrich/filter) → Processed Topic → Sink Connector → Data Warehouse/Lake. Key considerations: Use Schema Registry with BACKWARD compatibility for schema governance. Partition by business key (customer_id, order_id) for ordering. Monitor consumer lag. Use exactly_once_v2 in Kafka Streams for correctness. Sink connector handles backpressure. Dead Letter Queue for malformed records. Separate topics per entity type. Use log compaction for latest-state topics feeding dimensions.
What is KRaft and why does it matter?▼
KRaft (Kafka Raft) is Kafka's built-in metadata consensus protocol replacing ZooKeeper. Available production-ready since Kafka 3.3+. Benefits: simplified operations (one less system to manage), faster controller failover (<1s vs minutes), dramatically higher scalability (millions of partitions vs tens of thousands), self-describing metadata via internal __cluster_metadata topic. Node roles: controller (metadata quorum), broker (data), or combined (both — for smaller deployments). ZooKeeper still supported but is being deprecated.

Quick Reference

10 Golden Rules for Kafka
  • 01Partitions are the unit of parallelism — scale partitions to scale throughput
  • 02Same key = same partition = ordering guarantee per key
  • 03Max consumer parallelism per group = number of partitions
  • 04acks=all + min.insync.replicas=2 + enable.idempotence=true = production durability
  • 05Never decrease partition count — plan upfront or you'll regret it
  • 06Consumer lag is your most critical SLA metric — alert on sustained growth
  • 07KTable topics must use log.cleanup.policy=compact, not delete
  • 08ISR must have ≥ min.insync.replicas brokers or producers with acks=all will throw
  • 09Rebalances cause processing pauses — use cooperative rebalance + static membership
  • 10Schema Registry with BACKWARD_TRANSITIVE = safe rolling schema upgrades
