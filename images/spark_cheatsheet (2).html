Data Model & Execution Paths
How RDD · DataFrame · UDF · pandas_udf actually execute — and when to use each
INTERMEDIATE
Understanding how each API executes at the JVM level is the foundation for every optimization decision. The same logical operation can be 1× or 100× slower depending on which path you choose.
Path 1 — RDD: Java objects in heap, zero optimization
# RDD — operator-at-a-time, Java objects
rdd = sc.textFile("s3://orders/")
result = rdd \
    .map(lambda line: line.split(",")) \
    .filter(lambda r: r[2] == "completed") \
    .map(lambda r: (r[1], float(r[3]))) \
    .reduceByKey(lambda a,b: a+b)
What happens at the JVM level

Each row is a Python/Scala object on the JVM heap. Each operator (map, filter) is a separate function call per row. No schema — Spark has no idea what's inside. No Catalyst rewrites. No Tungsten codegen. Java/Kryo serialization at every shuffle. GC must track every row object.

Use RDD only when: custom binary serialization format, graph algorithms (GraphX), ML algorithms not in MLlib, legacy code you cannot rewrite. For everything else — DataFrame is faster.
Path 2 — DataFrame: UnsafeRow binary, Catalyst + Tungsten
# DataFrame — Catalyst rewrites, Tungsten executes
result = spark.read.parquet("s3://orders/") \
    .filter(col("status") == "completed") \
    .groupBy("country") \
    .agg(F.sum("amount"))
What happens at the JVM level

Rows stored as UnsafeRow flat byte arrays — off-heap, no GC. Catalyst pushes filter into Parquet scan (rows discarded at disk). Tungsten fuses filter+project+aggregate into one JVM method (no per-operator function calls). Shuffle uses zero-copy binary transfer. Result: 10–100× faster than equivalent RDD.

Always use DataFrame for structured/semi-structured data. Catalyst + Tungsten give you free optimizations you can't replicate manually.
Path 3 — Python row UDF: JVM↔Python pickle, codegen broken
# Row UDF — per-row serialization loop
from pyspark.sql.functions import udf
from pyspark.sql.types import StringType

@udf(StringType())
def mask_phone(phone):
    return phone[:3] + "****" + phone[-2:]

df.withColumn("masked", mask_phone(col("phone")))
What happens for EACH of 10M rows

① JVM picks UnsafeRow → ② serialize to pickle bytes → ③ send over localhost socket → ④ Python worker deserialize → ⑤ run your function → ⑥ Python serialize result → ⑦ send back over socket → ⑧ JVM deserialize. = 4 ser/deser + 2 socket calls per row. Also: entire stage's Tungsten codegen is disabled — every operator in the stage runs interpreted.

Cost: 10M rows × (4 ser/deser + 2 socket calls) = hundreds of millions of operations. Typical row UDF is 10–100× slower than an equivalent native function. Only use when no native function exists AND pandas_udf is not possible.
Path 4 — pandas_udf + Apache Arrow: columnar batches, fast
Apache Arrow is a cross-language, columnar in-memory data format. Instead of sending rows one-by-one as pickle bytes, Arrow sends entire columns as contiguous typed arrays. The JVM and Python both understand Arrow natively — zero conversion between them. This is what makes pandas_udf fast.
# pandas_udf — Arrow batches, vectorised
from pyspark.sql.functions import pandas_udf
import pandas as pd

@pandas_udf("string")
def mask_phone_vec(s: pd.Series) -> pd.Series:
    return s.str[:3] + "****" + s.str[-2:]

df.withColumn("masked", mask_phone_vec(col("phone")))
What happens with Arrow batches

JVM partitions a column of 10,000 values into an Arrow array → sends entire array across socket in ONE call → Python receives as pd.Series with zero copy → pandas vectorized operation runs on the whole series → Arrow array returned in ONE call. = 2 socket calls per 10,000 rows instead of 2 per row. 5,000× fewer socket calls. Codegen still broken for that stage, but data transfer cost drops dramatically.

# Arrow config — always enable for pandas_udf and toPandas()
spark.conf.set("spark.sql.execution.arrow.pyspark.enabled", "true")
spark.conf.set("spark.sql.execution.arrow.maxRecordsPerBatch", "10000")  # rows per Arrow batch

# toPandas() also uses Arrow when enabled — 10× faster than row-by-row collection
pdf = df.limit(100000).toPandas()  # fast with Arrow; still avoid on huge DFs

The core insight: Catalyst can only optimize what it can see. A native F.regexp_replace() is a transparent tree node — Catalyst can push it down, reorder it, fold it. Transformations vs Actions
Transformations (Lazy) — build the DAG, no execution.
Narrow: map, filter, select, withColumn, flatMap
Wide (shuffle): groupBy, join, distinct, repartition, window
Actions (Eager) — trigger full DAG execution.
collect(), count(), show(), write(), take(n), first(), save()
Rule: Narrow transforms keep partition count unchanged. Wide transforms reset it to spark.sql.shuffle.partitions (default 200 — almost always wrong for production data).

Jobs → Stages → Tasks lifecycle
INTERMEDIATE
Lifecycle: Code → Bytes on Disk
# Your code
df.filter(col("status")=="completed").groupBy("country").count()

# What Spark does:
# 1. Parsed Logical Plan   — unresolved tree of operations
# 2. Analyzed Logical Plan — columns resolved against schema
# 3. Optimized Logical Plan— Catalyst: predicate pushdown, column pruning, constant folding
# 4. Physical Plan          — join algorithm chosen, stages determined
# 5. DAG split at Exchange  — each Exchange node = one stage boundary
# 6. Tasks dispatched       — one task per partition per stage
Materialization
Before materialization
Spark holds a plan (DAG). Zero bytes exist. Zero computation done. Memory used: ~0.
Trigger (action)
Any action — count(), write(), show() — fires the DAG. Plan becomes real data.
After materialization
Real rows exist in executor RAM, disk, or storage. cache() keeps them in RAM for reuse.

Total tasks = Σ (partitions per stage) ← NOT partitions × stages

Query Optimization Engine
Catalyst · Tungsten · AQE — the three layers that make Spark fast
MASTER
These three engines work in sequence: Catalyst rewrites your logical query into an optimal plan at compile time → Tungsten executes that plan as fast as possible at runtime using off-heap memory and generated code → AQE re-optimizes the plan mid-execution using real shuffle statistics.
Catalyst Optimizer — 4-phase rule engine
# Phase 1: PARSING — SQL string / DataFrame API → unresolved logical tree
# df.filter(col("status")=="ok").groupBy("country").count()
# becomes: Aggregate[country] ← Filter[status=ok] ← UnresolvedRelation[df]
# Column names are strings — not yet checked against schema

# Phase 2: ANALYSIS — resolve column names, validate types, check schema
# UnresolvedAttribute("status") → AttributeReference("status", StringType, nullable=false)
# Throws AnalysisException if column doesn't exist or types mismatch

# Phase 3: LOGICAL OPTIMIZATION — rule-based rewrites on the logical tree
# Applied as a fixed-point loop: keep applying rules until plan stops changing
# Rules are pure tree transformations — no data touched

# Phase 4: PHYSICAL PLANNING — choose execution strategy
# Multiple physical plans generated, cost model picks the cheapest
# Decides: BroadcastHashJoin vs SortMergeJoin, HashAggregate vs SortAggregate
# Output: SparkPlan tree → handed to Tungsten for execution
Catalyst Rule Categories

Catalyst killers — things that stop optimization dead:
• Python row UDFs inside filter/join — opaque to Catalyst, predicate pushdown disabled at that node
• select("*") before filter — column pruning cannot see through wildcard
• Type mismatches on join keys — implicit cast inserted, prevents predicate pushdown
• Non-deterministic functions (rand(), uuid()) — cannot be folded or reordered safely

Tungsten Engine — how physical execution works
Tungsten is Spark's physical execution engine introduced in Spark 1.5. It bypasses the JVM object model entirely — operating on raw binary memory — and generates native Java bytecode for each query. It has three pillars:
1. UnsafeRow — off-heap binary format

Rows are stored as flat byte arrays, not Java objects. No object headers (16 bytes saved per object). No pointer chasing. GC never sees this memory — zero GC pressure from data. Enables zero-copy shuffle: data transferred directly without deserialization.

2. Whole-Stage Codegen (WSCG)

Multiple operators fused into a single JVM method per stage. Shown as *(N) prefix in explain(). Each row flows through filter→project→aggregate in a tight loop with no virtual dispatch. 2–10× faster than interpreted operator-at-a-time execution.

3. Cache-aware computation

Data structures designed around CPU cache line size (64 bytes). Sort buffers, hash maps, and aggregation tables fit in L2/L3 cache. Reduces main memory fetches — memory bandwidth is often the real bottleneck, not CPU.

# UnsafeRow memory layout — contrast with Java objects

# Java object (Row with 3 fields: String "US", Int 42, Double 1500.0)
# Object header:       16 bytes  (type pointer + lock word)
# String "US":         56 bytes  (object header + char array + length)
# Integer 42:          16 bytes  (object header + value)
# Double 1500.0:       16 bytes  (object header + value)
# ────────────────────────────────
# Total heap:          ~104 bytes for 3 simple fields + GC tracking overhead

# UnsafeRow binary layout (same 3 fields)
# Null bitmap:         8 bytes   (1 bit per field, padded to 8-byte word)
# Fixed-width section: 16 bytes  (Int 42 + Double 1500.0, 8 bytes each)
# Variable-length:     10 bytes  (offset+length pointer 8B + "US" data 2B)
# ────────────────────────────────
# Total bytes:         34 bytes — 3× smaller, zero GC, direct memory access
# Whole-Stage Codegen — what Spark generates for filter + project + aggregate
# Instead of: FilterOperator.process(row) → ProjectOperator.process(row) → AggOperator.process(row)
# Spark generates a SINGLE Java method like this:

# void processPartition(Iterator<UnsafeRow> input) {
#   HashMap aggMap = new HashMap();
#   while (input.hasNext()) {
#     UnsafeRow row = input.next();
#     if (row.getString(1).equals("completed")) {    // filter inlined
#       String country = row.getString(2);            // project inlined
#       aggMap.merge(country, 1L, Long::sum);         // aggregate inlined
#     }                                               // no virtual dispatch
#   }                                                 // no per-operator call overhead
# }

# See codegen output: df.explain("codegen")
Codegen breakers — operators that fall back to interpreted mode (no *(N) in explain):
• Python / Scala UDFs — opaque to the code generator
• Some complex window functions with unbounded frames
• SortAggregate fallback (when HashAggregate can't fit data)
• GenerateExec (explode/posexplode in some versions)

A single codegen-breaking operator doesn't just slow itself down — it breaks the entire stage's fusion, forcing every operator in that stage back to interpreted execution.
AQE — Adaptive Query Execution (Spark 3.0+)
Catalyst and Tungsten are static — the plan is fixed before execution starts. AQE makes the plan dynamic: Spark pauses after each shuffle, measures the actual output sizes, and re-optimizes the remaining plan using real data instead of estimates.
# Static planning (Catalyst alone) — plan is fixed at compile time
# Catalyst estimates: "orders JOIN customers will produce ~10 GB"
# Reality: filter reduces it to 200 MB → plan used wrong join strategy

# AQE re-planning loop
# ┌─────────────────────────────────────────────────────────────┐
# │  Stage 1 runs → produces shuffle files                      │
# │  AQE reads actual shuffle file sizes                        │
# │  "Stage 1 produced 180 MB, not the estimated 10 GB"         │
# │  AQE re-plans Stage 2:                                      │
# │    • Switches SortMergeJoin → BroadcastHashJoin (fits now!) │
# │    • Coalesces 200 shuffle partitions → 2 (most are empty)  │
# │    • Splits fat partition 17 (skewed) into 5 sub-tasks      │
# │  Stage 2 runs with the better plan                          │
# └─────────────────────────────────────────────────────────────┘
AQE Feature 1 — Coalescing small shuffle partitions
# Problem: set shuffle.partitions=1000 conservatively
# After stage 1, most partitions hold tiny amounts of data:
# Part 0: 12 MB  Part 1: 8 MB  Part 2: 130 MB  Part 3: 5 MB ...
# Without AQE: 1000 tasks created, 980 do almost nothing — waste

# With AQE: reads actual sizes, merges small adjacent partitions
# Part 0+1+3: merged into one 25 MB task (below advisory threshold)
# Part 2: stays alone at 130 MB (above threshold)
# Result: far fewer tasks, no empty-task waste

spark.conf.set("spark.sql.adaptive.coalescePartitions.enabled", "true")
spark.conf.set("spark.sql.adaptive.advisoryPartitionSizeInBytes", "134217728")  # 128 MB
spark.conf.set("spark.sql.adaptive.coalescePartitions.minPartitionNum", "1")
AQE Feature 2 — Dynamic join strategy switching
# Problem: Catalyst estimated orders JOIN customers → 8 GB
# Chose SortMergeJoin (2 shuffles) because estimate > broadcast threshold
# Reality: filter on orders runs first, result is only 150 MB

# Without AQE: still executes SortMergeJoin — 2 unnecessary shuffles
# With AQE: after Stage 1, sees 150 MB → switches to BroadcastHashJoin
# Saves 2 full shuffles — often the biggest single AQE win

spark.conf.set("spark.sql.adaptive.localShuffleReader.enabled", "true")

# AQE broadcast threshold (can be higher than static threshold — AQE knows real size)
spark.conf.set("spark.sql.adaptive.autoBroadcastJoinThreshold", "30m")
AQE Feature 3 — Skew join handling
# Problem: join key "US" has 60% of all rows → one partition 50 GB
# Others are ~200 MB → one task takes 45 min, rest finish in 2 min

# Without AQE: that one task is the straggler, whole stage blocked
# With AQE: detects partition 17 is 250× median size
#   → splits partition 17 into 5 sub-partitions
#   → replicates matching build-side data for each sub-partition
#   → 5 tasks of 10 GB each instead of 1 task of 50 GB
#   → stage completes 5× faster

spark.conf.set("spark.sql.adaptive.skewJoin.enabled", "true")
spark.conf.set("spark.sql.adaptive.skewJoin.skewedPartitionFactor", "5")       # 5× median
spark.conf.set("spark.sql.adaptive.skewJoin.skewedPartitionThresholdInBytes", "256m")

Shuffle Mechanics
The most expensive operation in Spark
ADVANCED
What triggers a shuffle
Always shuffles: groupBy, join (non-broadcast), distinct, repartition(), window functions without partition hints, orderBy on full dataset
Never shuffles: filter, map, select, withColumn, flatMap, coalesce (reduces only), union (if same schema)
Shuffle Internals (Sort-based, default since Spark 1.2)
# Map side (write):
# 1. Task computes its partition output
# 2. Rows hashed: hash(key) % numReducePartitions → bucket assignment
# 3. Spilled to local disk as sorted shuffle files (one per reducer bucket)
# 4. Driver notified: "Executor A has shuffle files for stage 3"

# Reduce side (read):
# 5. Reduce tasks start — fetch their bucket from ALL map tasks over network
# 6. Merge-sort fetched data (External sort if doesn't fit in memory)
# 7. Apply aggregate / join logic

# Cost = disk write (map) + network transfer + disk read (reduce) + deser
Shuffle cost ∝ data_size × (1/compression_ratio) × network_speed⁻¹
Shuffle Partition Sizing
numPartitions = total_shuffle_data_MB / target_partition_MB (100–200 MB)

Key Distribution & Hash Collision
partition_id = hash(key) % numPartitions — multiple keys land in same partition
N = distinct values
Perfect: 1 key per partition, 1 file handle per write task.
N < distinct values
Multiple keys collide per partition. Still correct, slightly more file handles.

SQL Best Practices
  • Use SQL hints (/*+ BROADCAST */) when Catalyst can't estimate size
  • Prefer approx_count_distinct over COUNT(DISTINCT) for dashboards
  • Use CTEs (WITH) for readability — same performance as subqueries
  • Always REFRESH TABLE after external writes to Hive tables
  • Use EXPLAIN in SQL: EXPLAIN EXTENDED SELECT ...

Core Concepts
Micro-batch (default)

Processes data in small batches at fixed intervals. Exactly-once semantics. Latency: 100ms–minutes. Good for most use cases.

Continuous Mode

~1ms latency. At-least-once only. Limited operations. Requires Kafka source. Experimental in most Spark versions.

Checkpointing

Writes offsets + state to HDFS/S3 after each batch. Enables exactly-once restarts. NEVER change checkpoint location mid-run.

# 4. Hyperparameter tuning

    .addGrid(gbt.maxDepth, [3, 5, 7]) \
    .addGrid(gbt.stepSize, [0.05, 0.1]) \
    .build()

# 5. Train / evaluate
train, test = df.randomSplit([0.8, 0.2], seed=42)
model = cv.fit(train)
auc   = evaluator.evaluate(model.transform(test))

MLlib Performance & Pitfalls
  • Checkpoint iterative algorithms every 10–20 iterations to break long DAG lineage
  • Cache training DF before fit() — MLlib reads it multiple times per iteration
  • Use parallelism param in CrossValidator to run folds in parallel
  • Persist model to S3/HDFS — save() is the only way to reuse across sessions
  • Use BinaryClassificationEvaluator (AUC) not accuracy for imbalanced data
  • Set seed on randomSplit AND model for reproducibility
  • Don't run CrossValidator without setting parallelism — sequential by default
  • Don't forget to fit scaler on train only — data leakage otherwise
  • Don't use MLlib for deep learning — use PyTorch/TF via TorchDistributor
  • Don't collect large prediction DFs — write to Delta, query downstream
  • Don't use StringIndexer without handleInvalid="keep" — will fail on new categories at predict time

Dynamic Allocation on K8s — shuffle tracking
# Shuffle Tracking (K8s mode) — how it works:
# Normal dynamic alloc: removes idle executors → their shuffle files are lost
# → downstream tasks that need those files fail
# Shuffle Tracking: Driver tracks which executors hold unread shuffle files
# → executor is NOT removed until all its shuffle output has been consumed
# → shuffle files never lost → no stage retries due to missing shuffle data

spark.conf.set("spark.dynamicAllocation.enabled", "true")
spark.conf.set("spark.dynamicAllocation.shuffleTracking.enabled", "true")
spark.conf.set("spark.dynamicAllocation.shuffleTracking.timeout", "600s")  # evict after 10 min even if unread
spark.conf.set("spark.dynamicAllocation.minExecutors", "2")
spark.conf.set("spark.dynamicAllocation.maxExecutors", "50")
spark.conf.set("spark.dynamicAllocation.executorIdleTimeout", "60s")
spark.conf.set("spark.dynamicAllocation.schedulerBacklogTimeout", "1s")

# ESS: separate long-running daemon holds ALL shuffle files → executor truly free to be removed
# Shuffle Tracking: no daemon needed → simpler K8s setup, but executor held longer

# 1. spark-submit parses flags + loads spark-defaults.conf
# 2. Uploads job JAR/Python file + dependencies to staging (HDFS/S3)
# 3. Contacts cluster manager (YARN RM / K8s API / Spark Master)
# 4. Cluster manager allocates container/pod for the Driver (AM on YARN)
# 5. Driver JVM starts, initialises SparkContext
# 6. Driver registers with cluster manager, requests executor resources
# 7. Cluster manager launches executor containers/pods
# 8. Executors register back with Driver via RPC
# 9. Driver submits tasks to executors via TaskScheduler
#10. On finish: executors deregister, Driver exits, resources released

Dynamic Allocation — the algorithm explained
# Scale-UP: tasks queued longer than schedulerBacklogTimeout (default 1s)
# Exponential: adds 1, then 2, then 4, then 8 executors per round — fast ramp-up
# Capped at maxExecutors

# Scale-DOWN: executor idle longer than executorIdleTimeout (default 60s)
# "Idle" = no tasks running AND no unread shuffle data (shuffle tracking check)
# Decommission: Driver sends signal → executor finishes current task → gracefully exits

# Without: job starts 0 executors → wait 1s → request 1 → wait for pod/container → cold start
# With initialExecutors=5: 5 executors ready immediately → job starts fast

Executor sizing formula & parallelism
MASTER
Executor Sizing Formula (on-premise HDFS)
Step 1: cores_per_executor = 5 (HDFS NIC saturation point)
Step 2: executors_per_node = floor(node_cores / 5) leave 1 core for OS
Step 3: ram_per_executor = (node_ram − 2 GB OS) / executors_per_node
Step 4: overhead = max(384 MB, 0.10 × ram_per_executor)
Step 5: executor.memory = ram_per_executor − overhead
Step 6: total_executors = executors_per_node × worker_nodes − 1 (for driver) 
# Example: 6 nodes × 16 cores × 64 GB
cores_per_executor   = 5
executors_per_node   = floor(16/5) = 3           # 1 core left for OS
ram_per_executor     = (64-2) / 3 = 20.7 GB
overhead             = max(384MB, 0.1×20.7GB) = 2.1 GB
executor_memory      = 20.7 - 2.1 = 18.6 ≈ 18g
total_executors      = 3 × 6 - 1 = 17
total_cores          = 17 × 5 = 85
shuffle_partitions   = max(3×85, data_GB×8) = max(255, data_GB×8)

Micro-batch execution internals — the StreamExecution loop
# Every trigger interval, Spark runs this loop:

# 1. OFFSET FETCH
#    Source.getOffset() — asks Kafka "what is the latest offset per partition?"
#    Result: {topic-0: 1542, topic-1: 987, topic-2: 2341}

# 2. OFFSET COMMIT to WAL (Write-Ahead Log)
#    Writes start + end offsets for this batch to checkpointLocation/_commits/
#    Done BEFORE processing — enables exactly-once on restart

# 3. BATCH PLANNING
#    Creates a bounded DataFrame from [lastOffset, newOffset] range
#    Catalyst compiles this exactly like a batch job — same physical plan

# 4. EXECUTION
#    Runs the batch DataFrame plan on executors
#    Tungsten, AQE, codegen — all apply exactly as in batch mode

# 5. SINK COMMIT
#    Writes output to sink (Delta, Kafka, files)
#    Idempotent sink ensures no duplicates even if this step retries

# 6. OFFSET CHECKPOINT
#    Writes committed offsets to checkpointLocation/offsets/
#    Only NOW are these offsets considered "processed"

# On restart after failure:
#    Read last committed offset from checkpoint → resume from that offset
#    Batch is replayed from where it stopped → exactly-once (with idempotent sink)
Never delete or move the checkpoint directory mid-run. The checkpoint holds the WAL, offset log, and state store. Deleting it forces a full restart from the beginning (or startingOffsets), potentially reprocessing millions of events or losing state entirely.
Exactly-once semantics — what it actually means
GuaranteeRequiresHow
Exactly-once processingReplayable source (Kafka, files)WAL records offsets before processing. On failure, replay from last committed offset — same data processed again.
Exactly-once outputIdempotent sink OR transactional sinkDelta Lake: batch ID embedded in transaction → duplicate write = no-op. Kafka: idempotent producer + exactly-once transactions.
At-least-once outputAny sinkProcessing is exactly-once but output may be written, then failure before checkpoint commit → retry writes same output again.

Watermarks & late data — the full picture
# Watermark = max(event_time seen so far) − delay
# Events with event_time < watermark are DROPPED
# State (window aggregates) older than watermark is CLEARED from memory

# Example: watermark delay = "10 minutes"
# Batch 1: max event_time = 12:05 → watermark = 11:55
# Batch 2: max event_time = 12:20 → watermark = 12:10
# Event arrives with event_time = 12:08 in batch 3
#   → 12:08 > 12:10? NO → event is LATE → DROPPED

# Multiple watermarks on a join — Spark uses the MINIMUM
# If stream A watermark = 12:30 and stream B watermark = 12:00
# Global watermark = 12:00 — state from A cannot be cleared until B catches up

# Choosing delay: balance freshness vs late data tolerance
# Too short: late events dropped, incomplete aggregates
# Too long:  state grows in memory, higher memory pressure
# Rule: set delay = P99 of your observed event_time lag from your data source

Stateful operations — mapGroupsWithState & flatMapGroupsWithState
Window aggregations cover most cases. When you need fully custom state — session tracking, fraud detection FSM, complex event processing — use these operators. State is stored in the State Store (RocksDB or in-memory) per group key, persisted to checkpoint on each batch.
# Use case: detect user sessions (gap > 30 min = new session)
from pyspark.sql.streaming.state import GroupStateTimeout
from pyspark.sql.types import *
import dataclasses

@dataclasses.dataclass
class SessionState:
    session_id: str
    start_time: float
    last_time:  float
    event_count: int

def update_session(user_id, events, state):
    if state.hasTimedOut():
        # 30-min gap — emit final session, clear state
        s = state.get
        state.remove()
        yield (user_id, s.session_id, s.start_time, s.last_time, s.event_count)
        return

    events_list = list(events)
    if state.exists:
        s = state.get
    else:
        s = SessionState(str(uuid4()), events_list[0].ts, events_list[0].ts, 0)

    s.last_time   = max(e.ts for e in events_list)
    s.event_count += len(events_list)
    state.update(s)
    state.setTimeoutDuration(30 * 60 * 1000)  # 30 min processing-time timeout

output_schema = "user_id string, session_id string, start double, end double, events int"

sessions = parsed \
    .groupBy("user_id") \
    .flatMapGroupsWithState(
        outputMode="append",
        timeoutConf=GroupStateTimeout.ProcessingTimeTimeout,
        func=update_session,
        outputStructType=output_schema,
        stateStructType="session_id string, start_time double, last_time double, event_count int"
    )

class MetricsListener(StreamingQueryListener):
    def onQueryStarted(self, event):
        print(f"Query started: {event.name} id={event.id}")

    def onQueryProgress(self, event):
        p = event.progress
        # Key metrics every production stream should track:
        lag           = p["sources"][0].get("endOffset", {})   # Kafka consumer lag
        input_rate    = p["inputRowsPerSecond"]
        process_rate  = p["processedRowsPerSecond"]
        batch_ms      = p["batchDuration"]
        state_rows    = p.get("stateOperators", [{}])[0].get("numRowsTotal", 0)

        # Alert: processing rate < input rate = falling behind
        if process_rate < input_rate * 0.9:
            alert(f"Stream {p['name']} falling behind: {process_rate:.0f} vs {input_rate:.0f} r/s")

        # Alert: batch takes longer than trigger interval = always catching up
        if batch_ms > 60000:  # 60s trigger interval
            alert(f"Batch duration {batch_ms}ms exceeds trigger interval")

        # Alert: state size growing unbounded
        if state_rows > 10_000_000:
            alert(f"State store has {state_rows:,} rows — watermark may be stuck")

        # Emit to Prometheus/DataDog
        metrics_client.gauge("spark.stream.input_rate",   input_rate,   tags=[p["name"]])
        metrics_client.gauge("spark.stream.process_rate", process_rate, tags=[p["name"]])
        metrics_client.gauge("spark.stream.batch_ms",     batch_ms,     tags=[p["name"]])
        metrics_client.gauge("spark.stream.state_rows",   state_rows,   tags=[p["name"]])

    def onQueryTerminated(self, event):
        if event.exception:
            alert(f"Query {event.name} terminated with exception: {event.exception}")

spark.streams.addListener(MetricsListener())

ProblemSymptomFix
Batch duration > trigger intervalStream always processing, never idle, consumer lag growsIncrease executors, set maxOffsetsPerTrigger to throttle input, optimize batch logic
State growing unboundednumRowsTotal in stateOperators keeps increasing, eventual OOMAdd watermark, reduce watermark delay, switch to RocksDB state store
Watermark not advancingState never cleared, memory grows, old windows never emittedCheck that event_time column is actually in the data. Late arriving events with far-future timestamps can advance watermark incorrectly.
Checkpoint corruption on schema changeStream fails on restart with schema mismatch after code changeSchema changes require new checkpoint location. Never change schema of stateful aggregations mid-run.
foreachBatch non-idempotentDuplicate rows in sink after retryUse txnVersion=batch_id on Delta, or deduplicate in sink using batch_id as idempotency key
Stream-stream join state explosionExecutor OOM, slow batchesAdd event_time range constraint to join condition. Ensures state bounded by time range, not number of unmatched rows.
failOnDataLoss=true (default)Stream crashes when Kafka retention deletes old offsetsSet failOnDataLoss=false. Log the gap and continue. Alternatively increase Kafka retention.
Production streaming checklist
  • Checkpoint on durable storage (S3/HDFS) — never local disk
  • Set maxOffsetsPerTrigger — prevents lag spike causing OOM batch
  • Use foreachBatch with txnVersion=batch_id for exactly-once Delta writes
  • Attach StreamingQueryListener — emit input/process rate and batch duration
  • Alert when processedRowsPerSecond < inputRowsPerSecond
  • Use RocksDB state store for stateful ops with >10M rows of state
  • Set availableNow=True trigger for scheduled batch-streaming hybrid jobs
  • Test watermark logic with MemoryStream source in unit tests
  • Don't change checkpoint location without starting fresh
  • Don't change stateful aggregation schema mid-run — restart from new checkpoint
  • Don't use complete output mode on large datasets — rewrites everything each batch
  • Don't omit watermark on stream-stream joins — unbounded state growth
  • Don't mix event_time and processing_time in the same watermark
  • Don't run streaming job without awaitTermination() — process exits immediately

Medallion Architecture (Delta Lake)

Bronze: raw ingestion, schema-on-read, append only.
Silver: cleaned, deduplicated, typed, joined.
Gold: aggregated, business-ready, query-optimized.
Each layer is a Delta table. MERGE for upserts. OPTIMIZE + VACUUM regularly.

Lambda Architecture

Batch layer: Spark batch on historical data, high accuracy.
Speed layer: Spark Streaming on recent data, low latency.
Serving layer: Merge batch + speed results at query time.
Being replaced by Kappa (streaming-only) + Delta Lake time travel.

Streaming Exactly-Once Pattern

Kafka source + Delta sink + checkpoint = exactly-once end-to-end.
Delta's idempotent writes + Kafka offset tracking = no duplicates even on restart.
Use trigger(availableNow=True) for scheduled micro-batch jobs.

Feature Store Pattern (ML)

Spark computes features offline → Delta table (offline store).
Point-in-time correct joins via Delta time travel.
Cache base_features DF used by multiple model pipelines.
Checkpoint iterative MLlib jobs to avoid lineage explosion.

from pyspark.sql import SparkSession
from delta.tables import DeltaTable

spark = SparkSession.builder \
    .appName("orders_etl") \
    .config("spark.sql.adaptive.enabled", "true") \
    .config("spark.sql.shuffle.partitions", "10000") \
    .config("spark.serializer", "org.apache.spark.serializer.KryoSerializer") \
    .getOrCreate()

# 1. READ — filter early for partition pruning
orders = spark.read.format("delta").load("s3://bronze/orders/") \
              .filter(col("date") == "2024-01-15")  # partition pruning

# 2. JOIN — broadcast small dimension table
products = spark.read.format("delta").load("s3://bronze/products/")
enriched = orders.join(broadcast(products), "product_id")

# 3. CACHE base if used in multiple branches
enriched.cache()
enriched.count()  # warm cache

# 4. TRANSFORM
silver = enriched \
    .filter(col("status").isin("completed", "shipped")) \
    .withColumn("revenue", col("qty") * col("unit_price")) \
    .withColumn("tier", F.when(col("revenue") > 1000, "high").otherwise("low"))

# 5. WRITE — repartition for clean output files
silver.repartition(50, col("country")) \
      .write.format("delta") \
      .mode("append") \
      .partitionBy("country", "date") \
      .save("s3://silver/orders/")

enriched.unpersist()  # release cache

Orchestration best practices
ConcernPattern
Secrets managementNever put credentials in DAG code or SparkConf. Use Airflow Connections/Variables, K8s Secrets with envFrom, or Vault agent injection via init containers.
Date injectionAlways pass execution date as an argument (--date {{ ds }}). Never derive the date inside the Spark job — makes backfills impossible.
IdempotencySpark jobs must be safe to re-run for the same date. Use write.mode("overwrite") or Delta MERGE. Airflow retries and manual re-runs depend on this.
SLA monitoringSet sla=timedelta(hours=2) on Airflow tasks or timeout on Argo sensors. Alert before the business deadline, not after it's missed.
Dependency injectionUse Airflow XComs or Argo output parameters to pass results between steps (e.g. row count from ETL to downstream quality check).
Airflow vs ArgoAirflow: richer UI, Python DAGs, larger ecosystem. Argo: pure K8s-native, YAML-based, better for K8s-heavy shops. Both work well with Spark Operator.
