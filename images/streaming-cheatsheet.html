Mental Models
How each framework thinks about data. Getting this right is the foundation of everything else.
FLINK The Infinite Pipeline

Think of Flink as a water pipe system. Events flow continuously through operators one at a time. The pipeline is always running — operators are persistent processes that react immediately to each event.

💡
No concept of "batches". Every record is processed the moment it arrives. State is held in memory/RocksDB between events.
Source
→
map()
→
keyBy()
→
window()
→
Sink

Latency: Sub-millisecond possible

Processing Speed: True per-event processing. Best for real-time critical latency. No micro-batch overhead.

SPARK The Infinite Table

Think of Spark Structured Streaming as a database table that grows forever. New data appends to the table; you query it with SQL/DataFrame APIs. Under the hood, Spark runs periodic micro-batches.

⚠
Processing is fundamentally micro-batch. Continuous mode exists but is experimental. Potential disk-based shuffles add latency.
Batch 1
→
Batch 2
→
Batch N…

Latency: ~100ms–seconds per trigger

Processing Speed: Micro-batch. Fixed intervals. Supports continuous mode (experimental) for ultra-low latency.

KAFKA STREAMS The Log as Database

Think of Kafka Streams as a smart consumer that transforms Kafka topics. It's an embedded library — no cluster, no server. Your app IS the stream processor. Kafka topics are the source of truth.

✓
Pure event-by-event processing. Relies entirely on Kafka for input/output — Kafka network latency is always a floor on your end-to-end latency.

Latency: Low but Kafka-bound

Processing Speed: Pure stream, per-event. But Kafka I/O adds extra latency vs Flink.

How each framework thinks about time
FLINK Time Philosophy
  • First-class event time with watermark propagation
  • Processing time as fallback
  • Per-record time tracking — minimal latency overhead
  • Watermarks flow through the DAG, pausing windows until caught up
SPARK Time Philosophy
  • Event time supported but within micro-batch windows
  • Watermark defined on DataFrame column
  • Late data handled at batch boundary
  • Less precise than Flink's per-record tracking
KAFKA STREAMS Time Philosophy
  • Event time via record timestamps
  • Wall-clock time via System.currentTimeMillis()
  • suppress() operator for final window results
  • Grace period for late data — silently drops beyond it
The Stream-Table Duality
ALL FRAMEWORKS Fundamental Duality

Stream → Table

Accumulate all events → you get a table (state at a point in time). E.g., counting events = a running total table.

// Kafka Streams stream.groupByKey() .count() // → KTable

Table → Stream

Every row change in a table can be emitted as a stream of change events (changelog stream).

// Kafka Streams table.toStream() .to("output-topic")

Why it matters

This duality is the foundation of joins, aggregations, and CDC. A stream is a diff log of a table. A table is a materialized view of a stream.

💡
Flink SQL, Spark SQL, and Kafka Streams KTable all exploit this duality.

Flink Architecture
FLINK Cluster Architecture

JobManager (Master)

  • JobGraph compilation & optimization
  • Task scheduling across TaskManagers
  • Checkpoint coordination
  • Failure recovery & restart
  • High-availability via ZooKeeper/Kubernetes
JobManager ├── Dispatcher // job submission ├── ResourceManager // slot management └── CheckpointCoordinator

  • Runs Task Slots (isolated execution units)
  • Each slot shares one JVM but separate classloaders

  • Manages local state (RocksDB or heap)
  • Reports heartbeat to JobManager
TaskManager ├── Slot 1 → [Source | Map | Sink] ├── Slot 2 → [Source | Map | Sink] └── Slot N → ...

Operator Chaining

Consecutive operators with same parallelism are chained into a single Task — no serialization overhead between them.

// Flink chains automatically: source → map → filter // becomes ONE task, zero network // keyBy() BREAKS the chain // (requires shuffle/network)
💡
Operator chaining is one of Flink's biggest performance advantages.

  • Runs StreamingQueryManager

Kafka Streams Architecture
KAFKA STREAMS Embedded Library Architecture

No Cluster Required

  • Runs inside your JVM application
  • Scales by adding more app instances
  • Kafka consumer groups handle partition assignment
  • Each instance processes assigned partitions
  • No cluster management overhead
ℹ
Scalability is bounded by Kafka partitions. Max parallelism = partition count.

Stream Thread Model

  • Each instance has N StreamThreads
  • Each thread processes one or more tasks
  • Task = one Kafka partition assignment
  • No mixing of CPU/network heavy tasks across threads
  • Static resource allocation model
KafkaStreams app ├── StreamThread-1 │ └── Task[0,0] → partition 0 ├── StreamThread-2 │ └── Task[0,1] → partition 1 └── StreamThread-N → ...

FLINK Trigger Strategies

  • ProcessingTimeTrigger — fires by wall clock

Backpressure
FLINK Backpressure

Credit-based flow control. Upstream operators slow down when downstream buffers fill. Propagates back through the entire pipeline. Dynamic throttling — producer automatically slows based on consumption rate.

✓
Flink's backpressure is seamless. Monitor it in the Flink Web UI — if > 0%, investigate the bottleneck operator.
SPARK Backpressure

Rate limiting via trigger interval. Spark limits Kafka fetch rate with maxOffsetsPerTrigger. Automatic rate adjustment with spark.streaming.backpressure.enabled.

⚠
Requires careful tuning of micro-batch sizes vs acceptable latency. Too small = overhead. Too large = late processing.
KAFKA STREAMS Backpressure

Pull-based model — no backpressure mechanism needed. Kafka Streams pulls records at its own pace. The side effect: slow processing causes lag to build up in intermediate Kafka topics, which can be a silent problem.

⚠
Monitor consumer lag on intermediate topics. Unbounded lag = your processor is falling behind.
Time & Event Ordering
The most nuanced part of stream processing. Getting time semantics wrong produces silently incorrect results.
Three Types of Time
Event Time

The timestamp embedded in the event itself — when something actually happened in the real world. Most correct for business logic.

✓
Use event time for anything where ordering matters: fraud detection, billing, user sessions, SLAs.
// Flink: assign event time stream.assignTimestampsAndWatermarks( WatermarkStrategy .forBoundedOutOfOrderness(Duration.ofSeconds(5)) .withTimestampAssigner( (e, t) -> e.getTimestamp() ) )
Processing Time

The wall-clock time when the system processes the event. Simplest to implement, but wrong if events arrive out of order or late.

⚠
Never use for correctness-sensitive aggregations. Results are non-deterministic on reprocessing.
// Flink: use processing time TumblingProcessingTimeWindows .of(Time.minutes(5)) // Spark: default if no watermark // Kafka Streams: System.currentTimeMillis()
Ingestion Time

When the event enters the streaming system (Kafka topic ingestion time). A compromise: more stable than processing time, but still not the true event time.

// Flink: ingestion time env.setStreamTimeCharacteristic( TimeCharacteristic.IngestionTime ) // Uses Kafka topic timestamp // Good for: Kafka-based pipelines // where producer timestamps are // close to true event time
Watermarks — The Core Mechanism
What is a Watermark?

A watermark is a statement: "I believe all events with timestamp ≤ W have been seen." It's the mechanism that lets a stream processor close a window and emit results, even when events can arrive late.

// Events arriving out of order: [t=100] [t=95] [t=110] [t=88] [t=105] ↑ Watermark = max_seen - lag = 110 - 5 = 105 // When watermark > window_end, // window fires and emits result
ℹ
Watermark is a tradeoff: higher lag = more late data captured = higher latency. Lower lag = faster results = more events missed.
Watermark Strategies
StrategyDescriptionUse When
forMonotonousTimestampsAssumes events arrive in orderOrdered sources
forBoundedOutOfOrdernessMax known delayKnown max lateness
Custom WatermarkGeneratorArbitrary logicComplex sources
Kafka source per-partitionPer-partition watermarks, mergedMulti-partition Kafka
⚠
In Kafka Streams, watermark is global across all partitions — the slowest partition blocks all windows. Kafka Streams' suppress() operator handles this.
Late Data Handling — Per Framework
FLINK Late Data — Most Comprehensive
  • Allowed Lateness: Keep window state open for extra time after watermark passes
  • Side Outputs: Route late events to a separate stream for special handling
  • Re-aggregation: Trigger window re-computation when late data arrives
  • Correct final results even with unexpected delays
// Flink: full late data handling OutputTag<Event> lateTag = new OutputTag<>("late"){}; stream .keyBy(e -> e.userId) .window(TumblingEventTimeWindows .of(Time.minutes(5))) .allowedLateness(Time.minutes(2)) .sideOutputLateData(lateTag) .sum("count");
SPARK Late Data — Watermark-Based
  • Watermark defined on event time column
  • Late data within watermark window is processed
  • Data outside watermark is silently dropped
  • No side output support for late data
  • Best-effort flow control, less precise than Flink
// Spark: watermark for late data df.withWatermark("eventTime", "10 minutes") .groupBy( window(col("eventTime"), "5 minutes"), col("userId") ).count()
⚠
Late data past the watermark is silently dropped with no way to recover it or route it elsewhere.
KAFKA STREAMS Late Data — Grace Period
  • Grace period defined per window
  • Late events within grace period are accepted
  • Events past grace period are silently dropped
  • Global watermark across all partitions (slowest partition controls)
  • Less control vs Flink's comprehensive approach
// Kafka Streams: grace period TimeWindows.ofSizeAndGrace( Duration.ofMinutes(5), // window Duration.ofMinutes(2) // grace )

  • Query the output sink (e.g., Delta table, JDBC)
  • No native queryable state API like Flink/Kafka Streams
⚠

Processing Semantics
Guarantees about what happens when things go wrong. The difference between at-most-once, at-least-once, and exactly-once can mean millions of dollars in a financial system.
The Three Guarantees
At-Most-Once

Events are processed 0 or 1 times. If a failure occurs, events may be lost. No retry on failure.

✗
Unacceptable for most production use cases. Only appropriate for metrics/logging where some loss is acceptable.

How: No checkpointing. Commit offsets before processing.

At-Least-Once

Events are processed 1 or more times. No data loss, but duplicates are possible on failure/recovery.

⚠
Acceptable if your downstream is idempotent (e.g., upserts by ID). Requires deduplication otherwise.

How: Checkpoint state + commit offsets after processing.

Exactly-Once

Each event affects the output exactly once, even on failure. The gold standard — hardest to achieve.

✓
Required for: financial transactions, inventory systems, billing, anything where duplicates = real-world consequences.

How: Distributed transactions, 2PC, or idempotent producers.

How Each Framework Achieves Exactly-Once
FLINK Exactly-Once A

Kafka → Kafka (most common)

  • Distributed snapshots (Chandy-Lamport algorithm)
  • Checkpoint barriers flow through the DAG
  • Kafka consumer offsets included in checkpoint
  • Kafka transactional producer for output
  • 2PC sink protocol for external sinks

External Sinks

  • Implements TwoPhaseCommitSinkFunction
  • Pre-commit on checkpoint, actual commit when checkpoint completes
  • Supports Kafka, JDBC, filesystems
✓
Most comprehensive exactly-once support of the three frameworks.
KAFKA STREAMS Exactly-Once B

Kafka → Kafka only

  • Exactly-once via Kafka's transactional API (EOS v1/v2)
  • Read-process-write atomic transactions
  • Offset + state store updates committed atomically

External Sinks

  • No built-in EOS for non-Kafka sinks
  • Must implement idempotency out-of-band
  • Higher overhead vs at-least-once
⚠
EOS v2 (KIP-447) reduces overhead significantly. Enable with processing.guarantee=exactly_once_v2.
SPARK Exactly-Once B

Within Spark (WAL-based)

  • Write-ahead log + checkpoint for state
  • Kafka source tracks offsets in checkpoint
  • Exactly-once processing within Spark's DAG

Output Guarantees

  • File sinks: Exactly-once via atomic commit
  • Kafka sink: Exactly-once via idempotent producer in foreachBatch
  • JDBC/others: Need idempotent writes (upsert by offset)
⚠
True end-to-end EOS requires carefully designed idempotent sinks. No native 2PC like Flink.
Windowing
Windows turn an infinite stream into finite chunks that can be aggregated. Choosing the right window type is critical for correctness.
Window Types
Tumbling Windows

Fixed-size, non-overlapping windows. Every event belongs to exactly one window.

// [0-5min] [5-10min] [10-15min] // Flink: .window(TumblingEventTimeWindows.of(Time.minutes(5))) // Spark: .groupBy(window(col("ts"), "5 minutes")) // Kafka Streams: TimeWindows.ofSizeWithNoGrace(Duration.ofMinutes(5))

Use for: Hourly/daily reports, fixed-interval metrics.

Sliding Windows

Fixed-size, overlapping windows. Events can belong to multiple windows.

// 10min window, slides every 5min // [0-10] [5-15] [10-20] ... // Flink: .window(SlidingEventTimeWindows.of( Time.minutes(10), Time.minutes(5))) // Spark: .groupBy(window(col("ts"), "10 minutes", "5 minutes")) // Kafka Streams: SlidingWindows.ofTimeDifferenceWithNoGrace( Duration.ofMinutes(10))

Use for: Moving averages, rolling metrics, anomaly detection windows.

Session Windows

Dynamic-size windows that close after a gap of inactivity. Perfect for user sessions.

// Close window after 30min idle // Flink: .window(EventTimeSessionWindows .withGap(Time.minutes(30))) // Kafka Streams: SessionWindows.ofInactivityGapWithNoGrace( Duration.ofMinutes(30)) // Spark: no native session windows // Use flatMapGroupsWithState instead

Use for: User sessions, clickstreams, IoT device activity.

FLINK Global Windows + Custom Triggers

One window per key, open forever. Requires a custom trigger to fire.

// Count-based window .window(GlobalWindows.create()) .trigger(CountTrigger.of(100)) .aggregate(...) // Or custom trigger logic .trigger(new MyCustomTrigger())

Use for: Count-based batching, custom business logic windows. Flink-only feature.

Windowing & Large State Cost
FLINK Large State Windowing A

With RocksDB backend, even terabytes of window state is manageable. Incremental checkpoints keep snapshot time constant regardless of state size.

  • Window state stored off-heap in RocksDB
  • Async incremental snapshots to S3/GCS
  • Restore from RocksDB snapshot directly — no replay
KAFKA STREAMS Large State Windowing B

RocksDB used for windowed stores, but recovery from changelog can be very slow for large state (must replay entire topic).

  • Standby replicas help but don't eliminate lag
  • No shared state between partitions = expensive for cross-partition windowing
  • Large state windowing is costly
SPARK Large State Windowing B

Default HDFS state store for windowing. RocksDB available as plugin. Write-ahead log can cause larger checkpoint sizes.

  • State locality not guaranteed (rebuilds on different executor)
  • Large windows = larger micro-batch processing time
  • RocksDB plugin (Databricks native) helps

Optimization
Performance tuning from basic configuration to advanced architectural patterns.
Flink Optimization
FLINK Parallelism & Slots
  • Set parallelism = number of Kafka partitions for source operators
  • Reduce parallelism at sinks if needed
  • Operator chaining avoids network overhead — don't break chains unnecessarily
  • Use disableChaining() only to debug bottlenecks

For external lookups (DB, cache), use Async I/O to avoid blocking the pipeline.

Fault Tolerance & Recovery
How each framework survives failures and recovers state. This directly impacts your RTO (Recovery Time Objective).
Checkpointing Mechanics
FLINK Chandy-Lamport Distributed Snapshots A
  • JobManager injects checkpoint barriers into source streams
  • Barriers flow through DAG with data records
  • Operator snapshots state when all input barriers arrive
  • Coordinator confirms checkpoint completion
  • Incremental (RocksDB): Only changed SST files uploaded
  • Recovery: Restore RocksDB snapshots directly from cloud — no replay
✓
With incremental checkpoints, snapshot time is constant regardless of state size. Recovery is fast.
KAFKA STREAMS Changelog + RocksDB B
  • Every state store write → appended to changelog Kafka topic
  • RocksDB is the local state store (persistent)
  • On failure: replay changelog from last committed offset
  • Long recovery for large state — must replay entire changelog
  • Standby replicas pre-build state on other instances for fast failover
  • Requires considerable engineering for very large state
⚠
Without standby replicas, recovery time scales linearly with state size. Plan accordingly.
SPARK WAL + Checkpoint B
  • Write-ahead log (WAL) for received data
  • Checkpoint writes state + offsets to HDFS/S3
  • State best-effort task locality — may rebuild on different executor
  • RocksDB plugin (non-Databricks: manual): snapshot checkpointing
  • Larger checkpoint sizes vs Flink incremental
  • Processing delays possible while state rebuilds
⚠
State locality is best-effort. After recovery, state may land on a different executor, causing a rebuild delay.

Debugging Checklist
Step-by-Step Debug Process

1. Identify the Bottleneck

  • Check Kafka consumer lag first
  • Flink: look for red operators in Web UI (backpressure)
  • Spark: compare batchDuration vs trigger interval
  • KStreams: check process-latency-avg per task

2. Check State Health

  • Is state growing unboundedly? → Add TTL
  • Are checkpoints growing each run? → State leak
  • RocksDB hit ratio low? → Increase block cache
  • Checkpoint duration > interval? → Reduce state or use incremental CP

3. Check Time & Watermarks

  • Is watermark advancing? → Check idle partitions
  • High late record rate? → Adjust watermark lag
  • Windows never closing? → Watermark stuck
  • Using event vs processing time correctly?

Decision Guide — When to Choose Which
CHOOSE FLINK WHEN
  • Latency must be sub-second (fraud detection, trading)
  • Complex event-time semantics with late data handling
  • State is very large (GB-TB) — use RocksDB backend
  • Need exactly-once end-to-end including external sinks
  • Complex joins: stream-stream, temporal joins
  • Custom windowing with complex triggers
  • Queryable state required
  • Team can invest in operational complexity
CHOOSE KAFKA STREAMS WHEN
  • Already deep in Kafka ecosystem
  • Building stateful microservices (interactive queries)
  • Want library, not a cluster to manage
  • Input/output is purely Kafka topics
  • Moderate throughput and complexity
  • Team is Java-native
  • Streaming topology is relatively simple
  • Want to scale by just adding instances
CHOOSE SPARK WHEN
  • Team already uses Spark for batch — reuse skills
  • Latency of 30s–minutes is acceptable
  • Heavy SQL transformations (Spark SQL is excellent)
  • Using Databricks or Delta Lake ecosystem
  • Batch + streaming unified on one platform
  • ML pipelines with MLlib integration
  • Lambda architecture: same code for batch and stream
  • High throughput, lower frequency jobs

Exactly-Once End-to-End Pipeline
FLINK True End-to-End EOS
// Flink EOS: Kafka → Flink → Kafka → JDBC // Step 1: Kafka source with EOS KafkaSource<Order> source = KafkaSource.<Order>builder() .setBootstrapServers(brokers) .setTopics("orders") .build(); // Step 2: Enable exactly-once checkpoint env.enableCheckpointing(60000, CheckpointingMode.EXACTLY_ONCE); // Step 3: Transactional Kafka sink KafkaSink<Result> sink = KafkaSink.<Result>builder() .setBootstrapServers(brokers) .setDeliveryGuarantee( DeliveryGuarantee.EXACTLY_ONCE) .setTransactionalIdPrefix("my-app") .build(); // Step 4: 2PC JDBC sink stream.addSink(new TwoPhaseCommitJDBCSink(...));
Idempotency Patterns for Sinks Without 2PC
  • Upsert by natural key: UPSERT INTO table ON CONFLICT(id) DO UPDATE
  • Offset-based dedup: Store (topic, partition, offset) as primary key
  • Idempotency token: Include message UUID, check-and-insert
  • Optimistic locking: Version field, reject stale writes
-- Idempotent upsert (PostgreSQL) INSERT INTO results (id, value, updated_at) VALUES (?, ?, NOW()) ON CONFLICT (id) DO UPDATE SET value = EXCLUDED.value, updated_at = EXCLUDED.updated_at WHERE results.updated_at < EXCLUDED.updated_at;
💡
The WHERE clause prevents older messages from overwriting newer ones during replay.

CDC & Schema Evolution
Change Data Capture (CDC) Patterns
// Debezium CDC → Flink/Kafka Streams // CDC event structure: { "op": "u", // c=create, u=update, d=delete "before": { ... }, // Previous state "after": { ... }, // New state "source": { "ts_ms": ..., "lsn": ... } } // Flink: native CDC connectors """ CREATE TABLE users ( id INT PRIMARY KEY NOT ENFORCED, name STRING, PRIMARY KEY (id) NOT ENFORCED ) WITH ('connector' = 'mysql-cdc', ...) """

Flink has native CDC source connectors (flink-cdc-connectors). Kafka Streams uses Debezium as a Kafka producer upstream.

Schema Evolution Strategies
  • Schema Registry (Confluent/Apicurio) — centralized schema versioning, backward/forward compatibility enforced at producer/consumer
  • Avro/Protobuf — binary formats with schema embedded; support field addition/removal with defaults
  • Flink Savepoints — take savepoint before changing state schema; use StateMigration interface
  • Kafka Streams — no built-in state migration; must dual-run old + new topology, migrate, cut over
  • Blue/Green deployment — run two versions in parallel, compare outputs before switching traffic
⚠
Never change a Kafka Streams topology without considering state migration. Adding/removing a state store without migration will fail on startup.
