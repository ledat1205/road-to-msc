Name _sources.yml / _models.yml with underscore prefix so they sort to top in editors. Use source_schema__object_name double-underscore convention.

Models

Beginner → Intermediate
Staging Model Pattern
sql/jinjastg_crm__customers.sql
{{ config(
    materialized = 'view',
    tags         = ['staging', 'crm']
) }}

WITH source AS (
    SELECT * FROM {{ source('crm', 'customers') }}
),

renamed AS (
    SELECT
        -- Primary key
        customer_id::VARCHAR    AS customer_id,

        -- Dimensions
        first_name              AS customer_first_name,
        last_name               AS customer_last_name,
        email::VARCHAR          AS customer_email,
        status::VARCHAR         AS customer_status,

        -- Timestamps
        created_at::TIMESTAMP   AS customer_created_at,
        updated_at::TIMESTAMP   AS customer_updated_at,

        -- Metadata
        _fivetran_deleted        AS is_deleted,
        _fivetran_synced         AS _loaded_at

    FROM source
    WHERE _fivetran_deleted = FALSE
)

SELECT * FROM renamed
Mart Model Pattern (Fact Table)
sql/jinjafct_orders.sql
{{ config(
    materialized  = 'table',
    cluster_by    = ['order_date'],
    tags          = ['mart', 'finance']
) }}

WITH orders AS (
    SELECT * FROM {{ ref('stg_payments__orders') }}
),

customers AS (
    SELECT * FROM {{ ref('dim_customers') }}
),

order_items AS (
    SELECT * FROM {{ ref('int_order_items_enriched') }}
),

final AS (
    SELECT
        o.order_id,
        o.customer_id,
        c.customer_segment,
        o.order_date,
        o.order_status,
        oi.total_items,
        oi.gross_revenue,
        oi.discount_amount,
        oi.gross_revenue - oi.discount_amount AS net_revenue,
        o.created_at
    FROM orders o
    LEFT JOIN customers c USING (customer_id)
    LEFT JOIN order_items oi USING (order_id)
)

SELECT * FROM final
Model YAML Documentation
yaml_finance__models.yml
version: 2

models:
  - name: fct_orders
    description: "One row per order. Source of truth for revenue reporting."
    meta:
      owner: "finance-team"
      tier: "gold"
    config:
      contract:
        enforced: true            # dbt 1.5+ model contracts
    columns:
      - name: order_id
        description: "Unique order identifier from payments system."
        data_type: varchar
        constraints:
          - type: not_null
          - type: unique
        tests:
          - not_null
          - unique
      - name: order_status
        description: "Current order lifecycle status."
        tests:
          - not_null
          - accepted_values:
              values: ['placed', 'shipped', 'delivered', 'returned']
      - name: net_revenue
        description: "Gross revenue minus discounts, in USD."
        data_type: numeric
        tests:
          - not_null
          - dbt_utils.expression_is_true:
              expression: ">= 0"

Sources & refs

Beginner
Source Definition
yaml_crm__sources.yml
version: 2

sources:
  - name: crm
    description: "Salesforce CRM via Fivetran"
    database: RAW
    schema: SALESFORCE
    loader: fivetran
    loaded_at_field: _fivetran_synced
    freshness:
      warn_after: {count: 12, period: hour}
      error_after: {count: 24, period: hour}
    tags: ['crm']
    tables:
      - name: customers
        description: "Customer accounts table"
        columns:
          - name: id
            tests: [unique, not_null]
      - name: opportunities
        freshness:           # override at table level
          warn_after: {count: 6, period: hour}
ref() vs source() vs var()
FunctionUseExample
source() Reference raw/ingested data {{ source('crm', 'customers') }}
ref() Reference another dbt model {{ ref('stg_crm__customers') }}
var() Project variable from dbt_project.yml {{ var('payment_methods') }}
env_var() Environment variable {{ env_var('WAREHOUSE') }}
this Current model (in incremental) {{ this.schema }}.{{ this.name }}
ℹ️
Always use {{ ref() }} instead of hard-coding schema.table. This builds the DAG automatically and enables cross-environment compilation.

Testing Strategy

Intermediate
Generic Tests (YAML)
yaml
columns:
  - name: order_id
    tests:
      - not_null
      - unique

  - name: status
    tests:
      - accepted_values:
          values: ['placed', 'shipped', 'delivered']
          severity: warn     # warn vs error
          where: "order_date >= '2023-01-01'"

  - name: customer_id
    tests:
      - relationships:
          to: ref('dim_customers')
          field: customer_id

# Model-level (multi-column)
tests:
  - dbt_utils.unique_combination_of_columns:
      combination_of_columns:
        - order_id
        - line_item_id
Singular Test (SQL)
sqltests/assert_net_revenue_non_negative.sql
-- Fails if any rows returned
SELECT
    order_id,
    net_revenue
FROM {{ ref('fct_orders') }}
WHERE
    net_revenue < 0
    AND order_status != 'returned'
Test Severity & Config
yaml
tests:
  - not_null:
      config:
        severity: warn       # error | warn
        warn_if: "!= 0"
        error_if: "> 100"
        limit: 50            # show up to N failures
        store_failures: true  # persist to table for debug
        where: "order_date >= current_date - 7"
💡
Set store_failures: true in production to capture failed test rows in a dbt_test__audit schema — makes debugging much faster.
Testing Pyramid — What to Test Where
Staging (Light)
  • Primary key: not_null + unique
  • Foreign key: relationships
  • Critical casts: not_null dates
Intermediate (Medium)
  • Grain assertion (unique combo)
  • Referential integrity
  • Row count sanity
Marts (Heavy)
  • All staging tests +
  • Accepted values (status enums)
  • Metric assertions (revenue ≥ 0)
  • Recency check
  • Custom singular tests

Materializations

Intermediate
Type When to use Pros Cons Config
view Staging models, lightweight transforms Zero storage, always fresh Re-computed on every query materialized: view
table Marts, BI-facing models, heavy joins Fast reads, pre-computed Full rebuild every run materialized: table
incremental Large event/fact tables, logs Only processes new data, fast Late-arriving data complexity materialized: incremental
ephemeral Intermediate CTEs, no DB object needed No DB overhead Not query-able directly, hard to debug materialized: ephemeral
materialized view Snowflake/BigQuery — refresh on query Auto-refresh, some warehouses Limited support, not all warehouses materialized: materialized_view

Incremental Models

Advanced
Basic Incremental Pattern
sql/jinjafct_events.sql
{{ config(
    materialized       = 'incremental',
    unique_key         = 'event_id',
    incremental_strategy = 'merge',
    on_schema_change   = 'append_new_columns',
    cluster_by         = ['event_date'],
    partition_by       = {
        'field': 'event_date',
        'data_type': 'date'
    }
) }}

WITH source_events AS (
    SELECT
        event_id,
        user_id,
        event_name,
        event_date,
        event_timestamp,
        properties
    FROM {{ source('analytics', 'events') }}

    {% if is_incremental() %}
    -- Only process new/updated rows
    WHERE event_timestamp >= (
        SELECT MAX(event_timestamp) - INTERVAL '3 hours'
        FROM {{ this }}
    )
    {% endif %}
)

SELECT * FROM source_events
Incremental Strategies Comparison
StrategyWarehouseUse When
append All Insert-only, immutable events
delete+insert All Partition replacement, no MERGE support
merge Snowflake, BigQuery, Redshift Upserts — rows can be updated
insert_overwrite BigQuery, Spark Partition-based overwrite
microbatch dbt 1.9+, all Time-bounded batches, late data
🚨
Always subtract a lookback window from MAX timestamp (e.g. - INTERVAL '3 hours') to catch late-arriving data. Pure MAX can silently miss rows.
on_schema_change Options
ValueBehavior
ignoreDefault — ignore schema changes
failRaise error if schema changed
append_new_columnsAdd new columns, keep old
sync_all_columnsAdd new, remove deleted columns

dbt Operations
jinjamacros/grant_select.sql
{% macro grant_select(role) %}
    {% set tables = dbt_utils.get_relations_by_pattern(
        schema_pattern = target.schema,
        table_pattern  = '%'
    ) %}

    {% for t in tables %}
        GRANT SELECT ON {{ t }} TO ROLE {{ role }};
    {% endfor %}
{% endmacro %}
dbt run-operation grant_select --args '{"role": "REPORTER"}'

Production Patterns & Best Practices

Production
CI/CD Pipeline (GitHub Actions)
yaml.github/workflows/dbt-ci.yml
on: [pull_request]

jobs:
  dbt-ci:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: pip install dbt-snowflake

      # Download prod artifacts for state comparison
      - run: dbt ls --target prod > /dev/null 2>&1 || true

      # Build only modified models + downstream
      - run: |
          dbt build \
            --select state:modified+ \
            --defer \
            --state ./prod-artifacts/ \
            --target ci \
            --profiles-dir .

      # Run tests on changed models
      - run: |
          dbt test \
            --select state:modified+ \
            --defer \
            --state ./prod-artifacts/
Slim CI — Defer Pattern
ℹ️
Slim CI uses --state + --defer to only run models that changed. For unmodified upstream deps, it reads results from prod artifacts — saving 90%+ of CI time & cost.

Custom Schema Pattern (generate_schema_name)
jinjamacros/generate_schema_name.sql
{% macro generate_schema_name(custom_schema_name, node) %}
    {% if target.name == 'prod' %}
        {# In prod: use custom schema name directly #}
        {% if custom_schema_name is not none %}
            {{ custom_schema_name | trim }}
        {% else %}
            {{ target.schema }}
        {% endif %}
    {% else %}
        {# In dev/ci: prefix with developer name #}
        {{ target.schema }}_{{ custom_schema_name | default('') }}
    {% endif %}
{% endmacro %}
Production Best Practices Checklist
📐 Modeling
  • One model per source table in staging
  • Double underscore: stg_source__table
  • No business logic in staging
  • All models have YAML + description
  • PK test on every model
  • Use ref() never hardcode table paths
  • Ephemeral for single-use CTEs only
🔒 Security & Ops
  • Key-based auth for service accounts
  • Secrets only via env_var()
  • Separate roles: DEV / PROD
  • GRANT in post-hook not manually
  • profiles.yml in .gitignore
  • Dedicated warehouse per environment
  • Log run metadata to audit table
🚀 Performance
  • Incremental for large tables (>1M rows)
  • cluster_by / partition_by on marts
  • Avoid SELECT * in marts
  • Use ephemeral not views for CTEs
  • Parallelism: tune threads per env
  • Trim unused columns in staging
  • Analyze / statistics after build
